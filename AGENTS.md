# Interview Transcription & Video Upload Flow

- **Entry points & routes**: Main upload UI lives at `src/pages/InterviewTranscriptions/UploadInterviewTranscription.tsx` (route `/interviews/upload`). Admin monitoring/list lives at `src/pages/InterviewTranscriptions/InterviewTranscriptionsPage.tsx` (`/interviews/progress-view` for regular users, `/interview-transcriptions` for admins). Detailed view with live chunks is `src/pages/InterviewTranscriptions/InterviewTranscriptionDetailsPage.tsx` (`/interview-transcriptions/:id`). From the HR connection card view (`src/components/interviews/InterviewsSection.tsx`) admins can also start transcription directly against an interview row.

- **Video upload flow**: Upload screens rely on `useResumableVideoUpload` (`src/hooks/useResumableVideoUpload.ts`). It streams a file to `VideosApi.uploadVideoResumable` (`src/api/videosApi.ts`) in chunked requests with `Content-Range`, keeping `Upload-Session-Id`/`Upload-Offset` to resume after partial (204) responses. Progress is tracked in hook state and exposed as `{status, progress.pct}`. Once server returns `VideoResponseDto`, UIs fetch richer metadata plus a signed playback URL through `VideosApi.getById` and render it via `VideoProcessingPreview` (`src/components/video/VideoProcessingPreview.tsx`), which normalizes backend phases (`receiving`, `hashing`, `uploading_s3`, `processing`, etc.) into `VideoPlayer` phases and shows helper text/actions to refresh.

- **Starting transcription**: After upload, `UploadInterviewTranscriptionPage` calls `InterviewTranscriptionsApi.start({ video_id })` and sends users to the progress list. The HR interview card action in `InterviewsSection` triggers the same `start` call and toasts the resulting transcription id. API surface is defined in `src/api/interviewTranscriptionsApi.ts` (`start`, `list`, `getById`, `getByVideoId`, `restart`), and statuses are `created | processing | restarted | done`.

- **Progress monitoring**: `InterviewTranscriptionsPage` lists requests with `TranscriptionStatusBadge` (`done` → secondary, others outlined). Admins can filter by user. Details page polls `getById` every 10s until `status === "done"` and lets admins restart via `restart` mutation, updating relevant react-query caches.

- **Live chunks & SSE**: `SseProvider` (`src/providers/SseProvider.tsx`) wraps the app (`src/App.tsx`) and connects to `${apiBaseURL}/sse/stream` with credentials. All events are buffered (up to 200) in context; custom listeners include `INTERVIEW_TRANSCRIPTION_EVENT`. `useInterviewTranscriptionsStream` (`src/hooks/useInterviewTranscriptionsStream.ts`) filters/validates envelopes into `InterviewTranscriptionMessage` objects and exposes them to the details page, which renders chunk bubbles until the backend marks the job `done`.

- **Expected SSE payload (per chunk and completion)**: The only custom SSE event we consume is `event: "interview-transcription-chunk"` with JSON payload `{ type: "interview-transcription-chunk", interviewTranscriptionId: string, videoId: string, chunkIndex: number, text: string, startTimeSec: number, endTimeSec: number, speakerLabel?: string }`. Chunks may arrive until the transcription finishes; there is no separate “done” SSE signal handled on the frontend—completion is detected when REST polling returns `status: "done"` (after which chunk rendering stops and the full text is fetched from `transcription_url`).
